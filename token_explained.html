Token：AI 讀文字前，會先把語言切成小積木

Token 是大型語言模型處理文字的基本單位。它不完全等於字、詞或中文字數，而是模型 tokenizer 依規則切出的片段；費用、上下文長度與輸出限制通常都用 token 計算。

Token 不是「一個字」

英文常見單字可能是一個 token，也可能被拆成字根或片段；中文有時一個字是一個 token，有時詞組會被一起切。標點、空白、換行與程式碼符號也可能佔 token。

因此同一段意思，換語言、換寫法、換模型，都可能消耗不同 token 數。

模型看到文字前，通常先經過這條管線

Tokenization 是把人類語言轉成模型可計算數字序列的橋樑。

1. 原始文字

使用者輸入、文件、程式碼或工具回傳。

2. Tokenizer 切分

依模型詞彙表把字串切成 token。

3. 轉成 ID

每個 token 對應一個數字 ID。

4. 模型計算

模型用這些 ID 預測下一個 token。

只要放進 prompt 或由模型輸出，通常都會納入 token 計算。

🔤 自然語言

中英文句子、標點、空白與換行都可能被計入。

💻 程式碼

縮排、括號、變數名與註解會大量增加 token。

📊 表格與 JSON

欄位名、引號、逗號、重複鍵值都會佔容量。

🧾 對話歷史

多輪對話越長，可用上下文預算越快被消耗。

Token budget 心智圖

輸入與輸出共用一個容量概念

如果模型有固定上下文上限，系統指令、使用者資料、工具結果與最終回答都要共享這個預算。輸入太長時，留給輸出的空間可能變少。

常見誤解

Token 少不一定代表內容少

高度重複、格式化良好的內容可能比較省；大量 JSON、表格、程式碼或罕見符號則可能比肉眼看起來更耗 token。實務上應用摘要、去重與只保留相關段落來控制成本。

Prompt 與文件處理的實務建議

理解 token 後，就能更有效設計 AI 工作流。

情境建議做法原因
長文件摘要先切章節，逐段摘要，再合併總結避免一次塞滿上下文並降低遺漏
程式碼分析提供相關檔案與錯誤輸出，不要整個 repo 盲貼程式碼符號與縮排很吃 token
多輪任務保留短決策紀錄與 TODO讓新一輪不用重新讀完整歷史
降低成本刪除重複表格、長 log、無關附件輸入與輸出 token 通常都會計費

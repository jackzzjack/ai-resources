Context Window：AI 當下能看見多少資訊

上下文視窗不是「記憶力」本身，而是模型在產生下一段回答時可同時攤開參考的工作桌大小。桌面越大，可處理的文件與對話越長；但也更需要整理與取捨。

Context window 就像模型的工作桌

系統指令、使用者問題、前面對話、貼上的文件、工具回傳，都要放在同一張桌上。模型回答時只能直接引用桌面上還看得到的東西；太早、太長或太雜的內容可能被擠出視野。

所以「上下文很大」代表能一次處理更多資料，但不代表模型一定會抓住重點。好的摘要、分段與任務邊界仍然很重要。

上下文裡通常包含哪些東西？

不是只有你剛打的那一句。模型會把多種訊息一起作為推理材料。

🧭 系統與開發者指令

規定模型角色、工具使用、安全邊界與輸出格式，是最優先的操作規則。

💬 對話歷史

前幾輪需求、修正與決策，讓模型維持任務連續性。

📄 文件與程式碼

貼上的文章、規格、錯誤訊息、程式片段，通常是最吃 token 的部分。

🛠️ 工具回傳

搜尋結果、檔案內容、API 回應、測試輸出，都會佔用上下文。

一次 AI 任務的上下文流動

上下文會隨著任務進行逐步累積，也可能因容量限制而被壓縮或截斷。

1. 任務進來

使用者提出問題或目標。

2. 加入規則

系統指令與工具限制先放入。

3. 讀取資料

文件、網頁或程式碼被加入上下文。

4. 模型推理

在可見資訊範圍內找關聯與答案。

5. 輸出/再循環

回答與新工具結果成為下一輪上下文。

管理上下文的 5 個習慣

先摘要長資料

把背景、需求、限制、決策壓成短版。

關鍵要求放最後

模型通常更容易遵守最新且明確的指令。

分段處理大型文件

先建立目錄與問題清單，再逐段分析。

保留決策紀錄

讓下一輪不必重新推理已確定的選項。

切換任務時重申目標

避免舊任務內容干擾新任務。

常見誤解

「放得下」不等於「一定用得好」

長上下文讓模型有機會看見更多資料，但如果資料順序混亂、指令互相衝突，或重點藏在大量雜訊中，回答品質仍會下降。把上下文整理成可讀的工作區，通常比單純追求更大容量更有效。

MCP 是什麼？
AI 連接工具的通用插座

Model Context Protocol 讓 AI 不只會聊天，也能用一致、可控的方式連接資料、服務與操作工具。

AI Agent工具呼叫新手圖解安全邊界
一眼看懂誰在合作一次請求為何需要它怎麼開始
TL;DR

把 MCP 想成「AI 的 USB-C 規格」

USB-C 不規定你接的是螢幕、硬碟或充電器，卻讓它們能以共同規格協作；MCP 也不綁定特定 AI 或服務，而是定義 AI 如何發現、理解與呼叫外部能力。

角色與關係

四個角色，各自負責一件事

先分清楚誰做判斷、誰負責轉接、誰提供能力，就不會把 MCP 誤認為模型或聊天機器人。

👤

使用者

提出目標，例如「整理本週會議並建立待辦」。

🧠

Host / AI 應用

承載模型與對話，決定是否、何時需要工具。

🔁

MCP Client

在應用內建立連線，按照協定向伺服器詢問可用能力。

🧰

MCP Server

把特定資料或服務包成可描述、可呼叫的工具與資源。

📚

Resources

供 AI 讀取的脈絡，例如文件、資料庫紀錄或檔案。

⚙️

Tools

會產生動作的能力，例如搜尋、建立事項、執行查詢。

沒有 MCP 與有 MCP，差在哪？

面向各自客製串接使用 MCP
工具描述每個應用自己定格式以共同協定描述能力與輸入
整合成本每新增一組服務就重做介面Host 與 Server 可各自擴充
可移植性常綁在單一產品或程式相容的應用可重用同類型 Server
安全設計散落在每段客製程式裡仍需權限與確認，但邊界更容易被清楚定義
先懂這個

MCP 不是「讓 AI 可以為所欲為」

常見誤解：接上就等於授權

不對。MCP 是溝通協定；能讀什麼、能改什麼，仍由 Server、帳號權限、應用設定與使用者確認共同決定。

你可以這樣想：門禁卡，不是萬能鑰匙

協定提供「刷卡方式」，但每一扇門是否開啟、可進哪些區域，仍由權限規則決定。

新手導讀

從概念走到實作，建議這樣看

① 先看使用情境：當你希望 AI 查閱指定文件、讀取資料庫、查詢內部服務，或執行受控操作時，MCP 是讓這些能力以一致形式提供給 AI 的方法。
② 再懂元件：記住最短鏈路：Host 裡的 Client 連到 Server；Server 提供 Resources、Tools 與提示資訊；AI 依目標選擇使用。
③ 最後管安全：從可信來源安裝 Server、給最小必要權限、對寫入／刪除／傳送等動作設確認點，並定期檢查可用工具清單。
